## Output 1 — Architecture Overview

### System Context
- **Monorepo**: 5 deployable units orchestrated via Docker Compose
  - `apps/main` — Next.js 14 (HR / Supervisor / Admin portal, port 3000)
  - `apps/portal` — Next.js 14 (Applicant-facing portal, port 3001)
  - `services/go-api` — Go 1.22 / Gin (REST API, port 8080)
  - `ai-service` — Python FastAPI (OCR + Embedding + Matching, port 8000)
  - `packages/shared` — Shared TypeScript types + design tokens (internal package)

### Infrastructure
- **PostgreSQL 15** — Primary data store (all migrations in `migrations/`)
- **Redis 7** — JWT blacklist, session store, pub/sub for notifications
- **Docker Compose** — Full service topology with health checks and dependency ordering

### Communication Pattern
- Frontend apps → Go API (REST over HTTP)
- Go API → AI Service (internal HTTP, async for OCR)
- Go API → PostgreSQL (pgx driver)
- Go API → Redis (session/JWT blacklist)
- Go API → SMTP (email via gomail)
- AI Service → Go API callback (OCR result webhook)

### Key Architectural Patterns
- **Action-based Pipeline**: HR selects actions freely — not a fixed linear flow
  - `Resume Review → Test and/or Interview (multiple rounds) → Hire / Drop / Transfer`
- **pipeline_actions table**: Central orchestration — one row per HR action, with round_number and phase tracking
- **Async OCR**: Resume upload triggers Go worker → FastAPI OCR → callback to Go API
- **Embedding Pipeline**: Job publish triggers JD+skill embedding via FastAPI; candidate resume OCR triggers resume embedding; match scores computed on-demand

## Output 6 — Architecture Decision Records (ADRs)

| ADR | Decision | Rationale |
|-----|----------|-----------|
| ADR-001 | M01+M02 share single auth module | Both use JWT + same user table; differentiate by role flag |
| ADR-002 | OCR is async via FastAPI queue | Must not block form submission; Go worker polls + callbacks |
| ADR-003 | Job scheduler is Go goroutine with DB polling | `publish_at` / `close_at` checked on interval; indexed columns for performance |
| ADR-004 | OCR status tracked in application_documents | `ocr_status` enum (pending/processing/done/failed) with index for worker polling |
| ADR-005 | pipeline_actions as central orchestration table | Action-based pipeline — HR selects freely, no fixed order; round_number + phase for multi-round support |
| ADR-006 | JSONB for embeddings instead of pgvector | Simpler deployment; pgvector migration path available when scale requires it |
| ADR-007 | Temp interviewers via invite_links + is_temp flag | 256-bit token, time-limited; user record created with is_temp=true on first access |
| ADR-008 | Form schema stored as JSON | Supports 14 field types dynamically; no table-per-field-type explosion |
| ADR-009 | Batch2 schemas for conflicting tables | job_embeddings (JSONB > pgvector for now), blacklist_logs (candidate_id + action enum), email_templates (code + type) |
