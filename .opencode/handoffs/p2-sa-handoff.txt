## System Overview
ATS Recruitment System — Monorepo with 2 apps:
- apps/main  — HR / Supervisor / Admin (Next.js frontend + Go backend)
- apps/portal — Applicant Portal (Next.js frontend + Go backend)
Both apps share one PostgreSQL database. No cross-app API calls — communication via shared DB only.
AI service: FastAPI (Python) — handles OCR, embedding, similarity computation.
Infrastructure: Docker Compose (dev), containerised deployment.

## Pipeline Model (Action-based)
HR selects actions freely per job. Mandatory order: Open → Resume Review. After that:
Test (N rounds) and/or Interview (N rounds) in any order → Pass / Drop / Transfer / Hire per applicant.
Transfer is only available during or after Interview.

## Roles
- Admin: user CRUD, password reset
- HR: full pipeline management, all decisions
- Supervisor: requisition creation, interviewer/grader assignment, Technical Test creation & grading
- Applicant: no account, portal access via URL only
- Temp Interviewer: temporary account auto-created via invite link, scoped to one round

## Existing DB Tables (names only — do NOT infer schema from names)
users, jobs, applications, candidates, forms, form_responses, test_rounds, interview_rounds,
interview_criteria, interview_scores, notifications, email_templates, master_data

## Key Gherkin AC Excerpts (SA-relevant constraints only)

### Auth (M01/M02)
- M01+M02 share one auth module; roles differentiated by flag (admin / hr / supervisor)
- Session timeout: 30 min inactivity
- OTP used only for HR self-service password reset (not admin-triggered)
- Forced password change on first login for all roles

### Applicant Portal (M03)
- No login required — applicant identified by email on submit
- Blacklist check: block submission if candidate email is blacklisted
- OCR: async, must not block form submission — enqueue to AI service queue
- Test link: re-openable until submit or expiry; auto-save answers

### Job Posting & Pipeline (M04)
- Job has: title, department, headcount, JD, hard skills (with weights), education, experience level, job type, default test(s), publish/close schedule
- Auto-publish / auto-close via background scheduler (Go worker or cron)
- Pipeline dashboard shows 5-stage milestone view + stats (Total/Remaining/Dropped/Hired)
- Email trigger available directly from pipeline dashboard

### Form Builder (M05)
- Form JSON must support 14 field types: Text, Textarea, Dropdown, Checkbox, Radio, FileUpload, DatePicker, MCQ, ShortAnswer, Essay, TrueFalse, RatingScale, Image/Video/Link
- Form delete blocked if referenced by active job
- Published form stored as JSON in DB; portal renders dynamically

### Resume Review (M06)
- 3-panel layout: left=applicant list, centre=resume viewer, right=personal info + LLM Summary + Position panel
- Overall Score = weighted avg of JD similarity + Hard Skill similarity (weights from Weight Popup)
- Interest action → applicant status: In Progress; Reject → Dropped (send SYS02 email); Star → Starred + Dropped from current job (NO email)
- Star button: applicant goes to Talent Pool; auto-rejected from current job
- Similarity pre-computed async on job publish for all starred candidates

### Action Test (M07)
- Round badge per applicant (increments per round)
- Phase 1 (Grading): real-time scores visible to HR as graders submit
- Phase 2 (Selection): HR manually triggers; leaderboard live-updates
- MCQ auto-graded; Essay/ShortAnswer go to Grading Inbox
- General Test graded by HR; Technical Test graded by owning Supervisor → FRNOT05 to HR
- Test link: same re-openable pattern as M03 (auto-save, blocked after submit)
- Anti-cheat: copy-paste blocked, tab-switch detected and recorded, random question order

### Action Interview (M08)
- Scoring criteria: criterion name + max_score per round; locked once any grader submits
- Interviewer types: Supervisor (existing user) + Temp (invite link → auto-created account)
- Temp account: 256-bit token URL, auto-deactivated when round closes or link expires/revoked
- Phase 1: per-interviewer scores real-time; HR sees who has not submitted
- Phase 2: avg-of-averages leaderboard (avg of each interviewer's per-criterion avg)
- Decision (Pass/Drop/Transfer/Hire) is final — no undo once confirmed
- Drop always triggers SYS02 rejection email

### Transfer (M09)
- Only available during or after Interview (Phase 1 or Phase 2)
- Source application closed, headcount slot returned; new Application created on target with history reference
- Unlimited sequential transfers supported
- If target position is Closed → warning + explicit HR confirmation required
- Transferred applicant excluded from all action lists on source position

### Talent Pool (M10)
- Starred candidate: auto-compute similarity vs all new jobs on publish (async)
- Blacklist removes Starred status; unblacklist does NOT restore it

### Candidate Management (M11)
- Candidate identified by email (unique key); created automatically on first application
- HR can edit name only
- Blacklist: mandatory reason, logs to Blacklist History (reason + timestamp + HR name)
- Unblacklist: mandatory reason, logs append (full history preserved)
- OCR: async trigger on resume upload → extract text + generate vector embedding

### Requisition (M12)
- Statuses: Draft → Pending → In-Progress → Published
- Supervisor cannot edit while Pending
- HR 1-click convert: pre-fills job form from requisition fields
- FRNOT01: new requisition → all HR; FRNOT02: job published (from requisition) → Supervisor

### Notifications (M13)
- 10 defined events: FRNOT01–FRNOT10
- Per-user channel toggle: In-App on/off + Email on/off per event type
- Default channels on first login: FRNOT03/09/10 = In-App only; all others = In-App + Email
- In-app: bell icon + unread badge + inbox panel

### Setup (M14)
- Email templates: SYS01/02/03 = read-only system templates; TPL = HR-managed custom templates
- Template delete blocked if in use by active job/action
- JD templates: pre-fill job form on select; delete removes from selection only
- Master data: Department, Employment Type, Experience Level, Hard Skill — reference-check on delete
- JD + Hard Skill embedding sent to AI service async after every job create/edit; weights stored with vectors

### Localization & Theme (M15)
- next-intl: all UI text via translation keys; th.json + en.json (nested namespaces)
- Language persisted: user profile (logged-in) / localStorage (applicant portal)
- Theme: CSS variables; follows OS preference on first load; persisted after user selection
- WCAG 2.1 AA: 4.5:1 contrast in both Light and Dark themes
- Brand: Vivid Red #F4001A (CTAs), Oxford Blue #0B1F3A (text), Plus Jakarta Sans
